## 🔴 CRITICAL

### C1. Authorization is missing on all notification routes
[src/app/api/notifications/route.ts](src/app/api/notifications/route.ts) — `GET`, `PATCH`,
`DELETE` accept `role`, `candidateId`, `targetRole` and notification `id` from
**query / body params** and never validate them against the logged-in
Supabase user.

Concrete impact (any authenticated session, candidate or HR):

- `GET /api/notifications?role=hr` → returns the HR-only feed even if caller is a candidate.
- `PATCH /api/notifications` with `{ id }` → marks **any** notification read.
- `PATCH /api/notifications` with `{ markAllRead: true, candidateId, targetRole: "HR" }` → forges actions on someone else's inbox.
- `DELETE /api/notifications?id=…` → deletes any notification by id.

RLS is disabled on every table (documented in CLAUDE.md), so there is no DB-level
backstop.

**Fix.** Resolve the caller via `createServerClient()` (`src/lib/supabase/server.ts`)
on every route, derive `role` from `user.user_metadata.role`, and look up the
candidate row by `user_id`. Reject if the body's `candidateId`/`id` doesn't match
the resolved identity (or the role is not HR for HR-only paths). The same audit
needs to be repeated for every `src/app/api/**/route.ts` — this file is the most
obvious offender, not the only one.

### C2. Onion violation — domain layer imports from client layer
[src/server/domain/services/job-requirements.schema.ts:19](src/server/domain/services/job-requirements.schema.ts#L19)

```ts
import { FIELDS_OF_WORK } from "@/client/lib/constants";
```

Domain must be dependency-free. This single import means any test of the domain
layer transitively pulls in the React-flavoured `src/client/lib/constants.ts`
module graph and any client-side polyfills it carries.

**Fix.** Move `FIELDS_OF_WORK` (and the matching type) to
`src/server/domain/value-objects/fields-of-work.ts`, then re-export it from
`src/client/lib/constants.ts` so existing UI imports keep working.

## 🟠 HIGH

### H1. PostgREST `.or()` filters are vulnerable to operator injection
Repos pass user input into `.or("col.ilike.%X%,col2.ilike.%X%")` after only
escaping single quotes — but the PostgREST `.or()` grammar is delimited by
**commas and dots**, not quotes. A search like `a,status.eq.HIRED` injects an
extra filter clause.

Hot spots:

- [src/server/infrastructure/database/candidate.repository.ts:54](src/server/infrastructure/database/candidate.repository.ts#L54) — `filters.search`
- [src/server/infrastructure/database/candidate.repository.ts:69](src/server/infrastructure/database/candidate.repository.ts#L69) — `filters.locationSearch`
- [src/server/infrastructure/database/job.repository.ts:36](src/server/infrastructure/database/job.repository.ts#L36) — search terms (Phase 5 dropped `country` from this list)
- [src/server/infrastructure/database/notification.repository.ts](src/server/infrastructure/database/notification.repository.ts) — multiple

**Fix.** Add a single sanitiser in
`src/server/infrastructure/database/db-utils.ts`:
```ts
export const escapeOrTerm = (s: string) =>
  s.replace(/[\\,()"]/g, "").replace(/'/g, "''").slice(0, 128);
```
Use it on every value before it touches `.or(...)`. Bonus: ban whitespace too if
the column is keyword-like.

### H2. Sequential `computeJobFit` over the entire candidate pool
[src/server/application/use-cases/job.use-cases.ts](src/server/application/use-cases/job.use-cases.ts)
`matchCandidatesToJob` loads **every** matchable candidate and iterates. Today's
demo dataset is small (~hundreds), but the `findForMatching()` query already
joins experiences/languages/education/skills, so payload grows linearly with
candidates × experiences and the route stalls noticeably for large jobs.

**Fix.** Pre-filter at the SQL layer using the JD's hard requirements:
required country (when present), at least one language match, has at least one
experience tagged with one of the JD's `fieldsOfWork`. The GIN index on
`experiences.fields_of_work` is already in place — use it
(`.contains()` / `.overlaps()`).

### H3. Two big features are stubs in production
- [src/app/dashboard/improvement/page.tsx](src/app/dashboard/improvement/page.tsx) —
  Improvement Tracks dashboard is empty even though `improvement_tracks` rows
  are auto-created on `FAIL` evaluations (Phase 5 in CLAUDE.md). Borderline
  candidates have no way to access their plan.
- [src/app/assess/[token]/page.tsx](src/app/assess/[token]/page.tsx) — magic-link
  written assessment surface is incomplete. The interview path is the only
  fully working assessment; HR cannot rely on the written variant.

**Fix.** Either ship them or hide the entry points (sidebar item, FAIL-side
notification copy) behind a feature flag until they're real.

### H4. `prisma/` directory is dead weight on disk
[prisma/schema.prisma](prisma/schema.prisma), [prisma/migrations/](prisma/migrations/)
The repo deleted Prisma in 2026-04-13 (CLAUDE.md "Migration Status" §6) but the
folder, schema, and ~10 migration directories remain. They confuse new devs
("which schema is canonical?") and bloat the working tree.

**Fix.** `git rm -r prisma/` after grepping for any leftover scripts referencing
it (none found in `package.json` scripts).

### H5. JSON Web Token / interview tokens validated only by HMAC
[src/server/infrastructure/auth/interview-token.service.ts] (and
`token_expires_at`/`signed_token_hash` in `interview_sessions`) — there is no
`jti`/replay protection: the same token can be re-used until expiry across
tabs/devices, and a leaked transcript URL grants the bearer full session
control. Acceptable for an academic project, but worth recording.

**Fix.** Bind the token to the candidate's session cookie (compare against the
`auth.users.id` resolved server-side), or single-use the token on first
`/interview/start`.

## 🟡 MEDIUM

### M1. ~84 `any` casts concentrated in three files
- [src/server/infrastructure/database/analytics.repository.ts](src/server/infrastructure/database/analytics.repository.ts) (~7 `(r: any) => …` mappings)
- [src/app/api/notifications/route.ts](src/app/api/notifications/route.ts) (~4)
- [src/server/application/use-cases/job.use-cases.ts](src/server/application/use-cases/job.use-cases.ts) (~6 around `matchCandidatesToJob`)

`camelizeKeys<any>(...)` returns `any` and propagates everywhere. The DB rows
have known shapes — define interfaces (`AnalyticsRow`, `RankedMatchRow`) and
parameterise `camelizeKeys<T>`.

### M2. Console logs in hot paths
- [src/server/infrastructure/ai/cv-parser.service.ts](src/server/infrastructure/ai/cv-parser.service.ts) (~lines 272–309) — logs partial LLM outputs
- [src/server/infrastructure/ai/openai-client.ts](src/server/infrastructure/ai/openai-client.ts)
- [src/server/infrastructure/scraping/adidas-job-scraper.service.ts](src/server/infrastructure/scraping/adidas-job-scraper.service.ts)

Vercel's logs aren't free; LLM outputs may contain PII from CVs. Replace with a
single `logger` wrapper that strips PII / can be silenced in production.

### M3. Candidate match-result UI still has an "everyone" universal score column
[src/app/dashboard/candidates/page.tsx](src/app/dashboard/candidates/page.tsx)
displays Quality (the CV-intrinsic profile score) and Fit (per-job). The
"Quality" name can mislead HR into ranking by it; CLAUDE.md is explicit it is
**not** a hiring signal.

**Fix.** Rename the column "Profile completeness" or add a tooltip with the
exact CLAUDE.md sentence: _"Useful as a prefilter, not a hiring signal."_

### M4. Comment drift after the Prisma → Supabase migration
Several `src/server/**` files still mention Prisma in comments
("Replaces PrismaXRepository" headers, `// Prisma raw query`). They are
historically accurate but obscure code reviews.

**Fix.** Single sweep replacing the Prisma references with Supabase context, or
delete them outright.

### M5. Tests don't cover authorisation
`tests/` has 134 passing tests but **none** assert authorisation:
- No test for the middleware redirect logic (`middleware.ts`).
- No test for `notifications` POST/PATCH/DELETE rejecting unauthenticated or
  cross-tenant calls.
- No test that `GET /api/jobs/[id]/match-candidates` is HR-only.

Given C1, this is the gap that allowed the auth bug to exist.

### M6. Domain-port return types are `Promise<any>`
[src/server/domain/ports/repositories.ts](src/server/domain/ports/repositories.ts)
`findById`, `findByExternalId`, `create`, `update`, `upsertByExternalId`
all return `Promise<any | null>` / `Promise<{ job: any; created: boolean }>`.
Domain ports are the single biggest type-safety opportunity — type the row
shape once at the port level and the rest of the stack inherits it for free.

### M7. `next.config.ts` is empty
[next.config.ts](next.config.ts) ships defaults. No image domain whitelist,
which means `next/image` on user-uploaded avatars or external job thumbnails
will eventually 500 once added. Document the intent or add `images.domains`.

  it (`pg_dump --schema-only` → `pg_to_dbml` or similar) and check it into
  `docs/`. Saves new contributors hours.
- **Structured logging.** Adopt `pino` (~10kB) once and route everything
  currently using `console.log` through it.
- **`vitest --coverage`.** Wire it into the GitHub Action so coverage trends
  are visible. The 134-test green is reassuring; the actual covered line %
  matters more.
- **Storybook for the design-system fragment** (`src/client/components/ui/*`).
  Useful for the demo + the academic write-up.

  small `parsing_jobs`-style table for JD parsing with `tokens_used`,
  `duration_ms`, and the raw response would surface drift when adidas changes
  their JD copy structure.
